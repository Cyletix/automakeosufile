## 结论先说

当前仓库和讨论里的目标方向 **部分对齐，但整体还停留在“原型验证”阶段**，还没有进入“可训练、可量化、可回溯调参”的架构。

如果按讨论里的路线继续推进，**现在最需要的不是直接训练模型，而是先把“数据入口 + 特征提取 + 特征质检 + GT 对齐评估”四件事做成稳定流水线**。

## 二、当前代码的实际架构状态

### 1. `main.py` 的角色：演示脚本，不是工程入口

`main.py` 当前更像一个 notebook 风格的实验脚本：

- 直接硬编码 `filename = "audio/NIGHTFALL.mp3"`
- 读取音频后临时算 STFT、Mel、Onset、Chroma
- 最后直接生成一个 `.osu`

问题在于：

- 没有参数配置层
- 没有模块边界
- 没有返回结构化结果
- 没有评估和调试输出
- 没有适配 `osu!mania` 训练数据入口

这意味着它适合“看看图长什么样”，不适合“做批量评估和训练”。

### 2. `fileprocess/osu_file_parse.py` 不能胜任训练标签解析

这个解析器对于 `.osu` 标准格式的处理是不完整的：

- `TimingPoints` 和 `HitObjects` 在 `.osu` 中本来就是逗号行，不是 `key:value`
- 当前实现用 `split(":")` 做通用解析，导致这两个 section 基本读不对
- 它也没有正确处理 mania 长条（hold note）字段

因此，**它目前不能作为训练标签解析器使用**。

### 3. `fileprocess/osu_file_make.py` 生成逻辑和目标模式不一致

当前生成器有几个关键问题：

- `Mode: 0`，这是标准模式，不是 `osu!mania`
- 所有 note 都写在中心点，实际上没有轨道映射
- 没有长条 note 生成
- 没有参考已有 `.osu` 的 timing points / offset / inherited timing

也就是说，**这个生成器更像“先写出一个能被 osu! 识别的最小文件”，而不是能和训练目标一致的 mania 谱面输出器。**

### 4. `密度修正专项/` 比主程序更接近“谱面后处理”

这里有两个值得保留的思路：

- `密度修正专项/OsuFile.py`
  - 能直接保留原始 hitobject 行
  - 能读取 `CircleSize` 作为键数
  - 能按轨道把 note 转成脉冲序列

- `密度修正专项/DensityFix.py`
  - 已经开始从“手感约束”角度处理谱面
  - 虽然规则还很粗糙，但方向是对的

这部分说明项目已经从“只看音频特征”开始转向“谱面可玩性修正”，和讨论里的后半段是接近的。

### 5. `temp_feat_old.py` 暗示了更正确的演进方向

这个文件虽然没有接进主程序，但里面已经出现了训练前需要的关键概念：

- BPM 检测
- 多级网格吸附
- 轨道映射
- 动态密度控制
- 物理手感约束

它的问题不是思路错，而是：

- 没有被整理成可运行模块
- 依赖的 `config` 不在当前主路径中
- 上游高质量特征还没有准备好

所以它更适合作为“下一版架构设计草稿”，而不是当前可直接运行的组件。

## 四、真实数据入口是否清楚

已经抽样确认了 `osu!` 标准目录关系：

- 歌曲根目录在 `D:/osu!/Songs`
- 一首歌曲通常对应一个文件夹
- 同目录下可同时存在：
  - 音频文件，如 `audio.mp3`
  - 多个 `.osu` 难度文件
  - 背景图等资源

抽样样本：`(sv)xi - Aragami`

- 目录内存在 `audio.mp3`
- `.osu` 文件头部 `[General]` 中明确写有 `AudioFilename: audio.mp3`
- `Mode: 3`，说明这是 `osu!mania`
- `CircleSize: 6`，说明是 6K 谱面

这说明你后续完全可以把“音频—谱面”自动配对起来做训练集。

### 对这个样本的判断

`Aragami` 这种高密度 6K 人工谱面适合做：

- 标准格式验证
- 高压特征测试
- 难图节奏吸附测试

但它**不适合作为第一首唯一基准曲**，因为：

- 密度高
- 节奏复杂
- 对 onset、BPM、lane mapping 的容错很低

更适合的策略是：

1. 用它做“压力测试样本”
2. 另外再选 1~2 首中低密度、结构更清晰的 4K/6K 谱面做“调参基准样本”

## 六、我建议你现在最需要的东西

如果只选一个最重要的当前任务，那就是：

## **先做“可验证的数据与特征基线”，而不是先做训练模型。**

更具体一点，就是下面这个顺序：

1. 先把 `osu!mania` 的音频—谱面配对读通
2. 再把 GT note/lane/hold 解析正确
3. 再把 STFT / Mel / Energy / Onset / CQT / Chroma 抽成统一接口
4. 先做质检和指标
5. 最后再讨论训练

原因很简单：

- 没有正确标签，训练无意义
- 没有高质量特征，训练只会学坏
- 没有指标，就不知道该改 onset 还是 lane mapping
